Deploying Hugo Site to Github Pages

  • June 12, 2023

This is a short writeup about publishing a Hugo site to github pages with two seperate repositories using Github workflows. I struggled finding a good explanation online when attempting to do this. Publishing this here in case it helps someone also struggling to get this working.

This writeup is intended to get you up and running quickly. I highly suggest you read the hugo docs

Step 1: Set Up

1. Install hugo

Follow the instructions from hugo docs to install hugo

2. Create a site

Navigate to the directory where you will be storing the repository. Open a git bash terminal and run the following command to create a new hugo site.

hugo new site linds-codes

replace linds-codes with a name for your source repo

Folder Structure should look like this:

  /Users/linds/sites/linds-codes/
  ├── archetypes
  ├── assets
  ├── data
  ├── layouts
  ├── resources
  ├── static
  ├── themes
  └── hugo.toml

Breakdown of the hugo directory structure here

This will be the source repository. Later on, we will use hugo to generate the /public folder. /public will generate the folders and files needed for the actual github pages site.

Initalize the repository:

  git init

3. Choose a Theme

Find a theme from the official list

I chose the Anatole theme.

Click download to be redirected the theme’s github repository.
The theme will have a readme or docs, with instructions on how to use the theme.

FORKING THE THEME:
To prevent headaches with git if you want to edit the base theme directly, fork the theme repository.

Get the url of your fork of the theme. My fork of anatole is located at: https://github.com/lindsay-kislingbury/anatole

We want to use the theme as a git submodule.
cd into the themes folder

  cd themes

Run the following to add the theme as a git submodule

  git submodule add https://github.com/lindsay-kislingbury/anatole

replace the url with the url of your fork of the theme’s repository

4. Setting Up Site

Normally, you will need to set the theme in the hugo.toml file.

  theme = anatole

However, when using the Anatola theme, I simply copied and pasted the entire config folder from the example site into the root of the repository. I then deleted the boilerplate hugo.toml.

I’m assuming that you are brand new to Hugo, and I suggest finding the config in the demo or example site and using that.

IMPORTANT: When using the theme as a submodule, do not use quotes around the theme name.

5. Add Some Content

To get up and running quickly, copy and paste the content from the example site into your content folder.

You can add, edit delete this content as necessary.

Now you can run

  hugo server

Navigate to localhost:1313 to see your site hosted locally.

6. Deploy

Run hugo to build the static site.

  hugo

A ./public folder will have been created in the repo’s directory. This contains everything we want to publish to github pages.

7. Create github Pages repo

Create a new repository on github for your site. The repo must follow this naming convention

6. Set Up Personal Access Token

Follow these instructions from github docs to create a Personal Access Token (PAT)

IMPORTANT name the token GH_PAGES

IMPORTANT select workflow and read: packages and write: packages

Copy the generated token.

Navigate to your source repo and add the token
settings -> secretes and variables -> actions -> new repository secret

Paste the PAT token and name the token GH_PAGES

7. Create a workflow

The purpose of the workflow is push the content of the ./public folder to a github pages site.

Create the following in the root of your repo:

  .github/workflows/main.yml

Folder structure shoud look like this:

  /Users/linds/sites/linds-codes/
  ├── .github/
  │   └── workflows/
  │       └── main.yml
  ├── archetypes
  ├── assets
  ├── data
  ├── layouts
  ├── resources
  ├── static
  ├── themes
  └── hugo.toml

Copy and paste the following into main.yml

# Workflow to help you get started with Hugo deployment

name: Deploy

# Controls when the action will run. Triggers the workflow on push or pull request
# events but only for the master branch
on:
  push:
    branches: master

# A workflow run is made up of one or more jobs that can run sequentially or in parallel
jobs:
  # This workflow contains a single job called "deploy"
  deploy:
    # The type of runner that the job will run on
    runs-on: ubuntu-latest

    # Steps represent a sequence of tasks that will be executed as part of the job
    steps:
      # Checks-out your repository under $GITHUB_WORKSPACE, so your job can access it
      - uses: actions/checkout@v2
      
      # Github Actions fo Hugo
      - name: Setup Hugo
        uses: peaceiris/actions-hugo@v2
        with:
          hugo-version: "0.113.0"
      
      # Creates ./public directory
      - name: Build
        run: hugo --minify
      
      # Upload/Commit the contents from publish_dir -> external_repository
      - name: Deploy
        uses: peaceiris/actions-gh-pages@v3
        with:
          external_repository: GITHUB-USERNAME/GITHUB-USERNAME.github.io
          publish_branch: master
          personal_token: ${{secrets.GH_TOKEN}}
          user_name: GITHUB-USERNAME-HERE
          user_email: GITHUB-EMAIL-HERE
          publish_dir: ./public

You need to edit the following fields with your details:
external_repository , user_name, user_email

7. Push Changes on watch the magic! 🔮

Push your changes in the source repo. Go to yourusername.github.io.

Navigate to the source repo Actions tab You should see the workflow execute

You should see the hugo site that you deployed in step 6!

Important Notes

  • After making changes to the source repo, when you want to see those changes reflected on github pages, you need to run hugo to build the files in ./public and then push.

  • When your PAT expires, you will need to generate a new PAT and change the Secret in your source repo. It is possible to create a PAT that never expires, but not reccomended.
